# Mappings were deduced from the Music Gaiji Dictionary for OS 8's
# Kotoeri input method.  Japanese names were extracted (and are given
# verbatim between square brackets), as were the Shift JIS character
# values.  The corresponding Unicode characters were generally
# selected to be as close a match as reasonably possible, using the
# bitmap font images provided by Enfour at
# http://www.enfour.co.jp/media/music/MusicBitmap.html as a guideline
# together with the character names extracted, as the font image
# appears to list characters in order when compared with the extracted
# names.

# Note that these mappings are not official (I have not used the
# official standard as a reference); although it is unlikely, given
# the inability to conclusively compare against a commercial CD-Text
# or MD encoding, it is possible that some of these mappings are not
# valid.  The JIS 0208X:1994 standard is assumed to hold for all
# characters outside of the 0xF640-0x???? block, even though there is
# no direct evidence for this.  Furthermore, due to the presence of
# identical character names for some characters, it is possible that
# certain characters may be misidentified (this is particularly true
# of Latin characters).

# Given the application of this encoding to strictly double-byte
# scenarios (e.g. as provided for in CD-Text), it seems plausible that
# no single-byte characters are valid, although this cannot be shown
# conclusively (this would, however, explain the presence of
# half-width Latin and Katakana characters that would otherwise be
# encoded using single-bytes).  We provide the standard single-byte
# translations of Shift JIS in any case, and implementers should be
# prepared to accept double-byte versions of the single-byte
# characters as padded with a preceding null byte in case this
# assumption proves false.  Round-tripping against this encoding
# should probably avoid encoding such values, and prefer the
# half-width variations in the 0xF640-0x???? block.

# Several rough blocks may be identified in the character set, as
# follows:

# The block from 0xF640 to 0xF6A0 very closely mirrors the ISO 8859-1
# character set (with a few exceptions), with characters that appear
# to be full-width.  Given the lack of similar characters elsewhere in
# these private-use mappings, and the presence of a complete
# half-width set elsewhere, it is assumed that these characters are
# meant to be full-width characters to match the full-width ASCII
# character set normally mapped from 0x00A0 to 0x00FF in standard
# Shift JIS.

# 0xF64B and 0xF65B were selected to be the LEFT-POINTING DOUBLE ANGLE
# QUOTATION MARK and RIGHT-POINTING DOUBLE ANGLE QUOTATION MARK
# respectively, rather than 0x226A MUCH LESS-THAN and 0x226B MUCH
# GREATER-THAN, which would be supported by the Japanese names; this
# was done given the close parallels of the sequence with ISO 8859-1,
# which encodes the former rather than the latter.

# We additionally add a mapping for 0x0909 to 0x0009 to correctly map
# the TAB character used by CD-Text (which uses 0x0909 instead of
# 0x0009 for double-byte character sets)

# DASH SYMBOL or ANGER SYMBOL?

# The first column contains the double-byte value in MS-JIS, the
# second contains a sequence of characters (of the form
# 0x####[+####...]) which we consider to be the mapping into Unicode.

# Some of the characters include 0xE01EE VARIATION SELECTOR-255 or
# 0xE01EF VARIATION SELECTOR-256 as part of the Unicode character
# mapping (usually to denote the opposite width of that usually
# assumed to hold, such as half-width Hiragana, or full-width ISO
# 8859-1 characters).  Although any character sequence including this
# character is necessarily invalid Unicode, the behavior of these
# characters (unlike other modifier characters) is default ignorable,
# and the relative unlikelihood of their use in approved variation
# sequences means that they are reasonably safe for use privately for
# round-trip purposes and, while, in compliant Unicode
# implementations, they will be ignored for display and sorting
# purposes.

# If Unicode-compliant round-tripping is desired, these sequences may
# be instead replaced with a character in the Private-Use Area.  It is
# suggested that the characters be mapped according to the following
# algorithm which is compatible with Microsoft's handling of EUDC
# characters (SAP and ICU support this same mapping):

# F040-F07E -> E000-E03E
# F080-F0FC -> E03F-E0BB
# F140-F17E -> E0BC-E0FA
# F180-F1FC -> E0FB-E177
# ...
# F940-F97E -> E69C-E6DA
# F980-F9FC -> E6DB-E757

# If correctness is desired without round-trip capabilities, the
# variation selector may alternately be left off.

# Despite the attempt to not map to the Private-Use Area, several
# characters have no readily identifiable Unicode counterpart.
# Specifically:

# 0xF75A  MINOR CHORD [まいなーこーど]
# 0xF75B  MAJOR CHORD [めじゃーこーど]
# 0xF75C  SUSPENDED CHORD [さすこーど]
# 0xF75D  AUGMENTED CHORD [おーぐめんと]
# 0xF75E  DIMINISHED CHORD [でぃみにっしゅ]
# 0xF75F  ADDED TONE CHORD [あどこーど]
# 0xF762  TERA MARK [てらまーく]
# 0xF7A8  COPY [こぴー]
# 0xF7A9  KEEP [きーぷ]
# 0xF7AA  MASTER [ますたー]
# 0xF7C6  TEAR [なみだ]
# 0xF7CF  SEA [うみ]
# 0xF7D0  MOUNTAIN [やま]

# according to the Shift-JIS mapping discussed above.
